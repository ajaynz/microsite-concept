Market Signal

Everyone is racing to build a software factory. Not buying one. Building one

A factory is infrastructure. It’s process. It’s a system that runs reliably at scale. And increasingly, autonomous coding agents that generate code, review, test, and merge pull requests around the clock.

Minions merges over 1,300 AI generated pull requests every week. Background coding agents go from a Slack message to a CI-passing PR with no human in the loop. Engineers tag the Minion bot in Slack and walk away. The agent runs in an isolated pre-warmed devbox, executes the task, and opens a PR.

Honk has shipped more than 1,500 agent-authored PRs into production. Spotify adapted its fleet management system into a platform for running a background coding agent that performs large-scale code migrations from natural language instructions.

Inspect now authors roughly 30% of Ramp’s merged pull requests. It runs inside sandboxed cloud VMs with full access to the real development environment, from Postgres and Redis to Temporal, LaunchDarkly, and Buildkite.

Minion has 84% of Uber developers using agentic tooling monthly. Uber also built an internal MCP gateway that exposes source code, Jira, Slack, and engineering documentation as a unified context surface for agents.

SOFTWARE FACTORY CAN'T FUNCTION WITHOUT RIGHT CONTEXT

Every serious agent team keeps rediscovering the same failure mode.

01

The failure is usually not “the model is weak.” The recurring problem is that the agent cannot find the right definitions, callers, history, conventions, or downstream dependencies in time.

02

Prompting humans to hand-carry context does not scale. If users have to guess the right grep commands or manually paste the relevant files, the factory depends on a human preprocessor, not a reliable context system.

03

Bad retrieval fails quietly. Too much context wastes reasoning budget. Too little context produces plausible output that feels credible but misses the actual implementation and blast radius.

04

Most benchmarks barely test this layer. Self-contained coding tasks hide the real retrieval problem. Cross-repository, multi-service work is where weak context surfaces are exposed.

Does the agent have the correct context?

A good answer depends on more than file access. It depends on reaching the real implementation, understanding impact, pulling the right history, and showing its work.

01

Can it find the canonical implementation?

If a symbol is implemented in a shared service or hidden behind an interface, the agent should still land on the real source of truth instead of a nearby lookalike.

Find the actual source of truth before reasoning starts.

02

Can it trace blast radius across repositories?

The agent should know who calls the thing being changed, which consumers are downstream, and where the migration risk actually sits.

File access alone stops being enough at repository boundaries.

03

Can it explain the answer with evidence?

A trustworthy output points back to paths, symbols, references, or history. It should feel inspectable, not magical.

A reviewer should be able to verify how the agent knew what it knew.

04

Can it bring in history and conventions without flooding the window?

Retrieval quality is precision, not volume. The system should surface the few artifacts that explain prior intent, ownership, and the existing team pattern.

Too much context is another form of failure.
